# holmes_area_direct 現行仕様

## 1. この文書の位置づけ

この文書は、`holmes_area_direct` の現行実装を、AI と人間の両方が 1 本で理解するための正本である。

この文書が説明する対象は次のとおり。

- 画面上で何ができるか
- 完全JSONの正規構造
- 各キーが何のためにあり、実行時にどう使われるか
- カテゴリJSONの受理形式とマージ規則
- 条件判定と effects の正確な意味
- event の見え方、遷移、終了条件
- サンプルデータの所属ルール

これは理想設計書ではない。  
`index.html`、`assets/js/app.js`、`assets/js/data/*.js`、`assets/js/prompts/*.js` の現在の実装に合わせて書いている。

## 2. ツール全体の役割

`holmes_area_direct` は、1 つの画面の中に次の 2 つの役割を持つ。

### 2.1 サウンドノベル JSON ランナー

完全JSONを読み込み、現在位置、現在 event、flags、inventory、time、quest 状態などを使って進行する。

このランナーでできること:

- spot 単位で探索を進める
- 同じ spot に複数 event を置き、条件で出し分ける
- `conditional_texts` で再訪時の本文を変える
- `actions` と `choices` を使い分ける
- flags、inventory、NPC 位置、state 数値、unique 数値で分岐する
- ending と game_over を event として扱う
- scenario の切替、quest の開始、主クエスト切替、quest 完了を effects で制御する

### 2.2 オーサリング補助ツール

完全JSONを直接書くだけでなく、カテゴリ別編集と AI 往復を支援する。

この補助ツールでできること:

- 基本エディタでカテゴリ単位に作成・編集・複製・削除する
- 現在カテゴリだけをカテゴリJSONとして切り出す
- AI にカテゴリJSON生成を依頼するための prompt をコピーする
- AI が返したカテゴリJSONを確認してからマージする
- 完全JSON欄とカテゴリJSON欄を別々に管理する
- どちらの欄も、ファイル入出力とクリップボード入出力を個別に行う

## 3. 画面構成と各カードでできること

画面は次のカードで構成される。

- 通知
- セットアップ
- エディタ
- AI支援
- 進行
- 状態
- 本文
- ログ

### 3.1 通知カード

直近の結果や失敗理由を表示する場所である。

ここに表示されるもの:

- JSON parse 失敗
- validate 失敗
- カテゴリJSONマージ失敗
- コピー成功 / 失敗
- アップロード成功 / 失敗
- ダウンロード成功 / 失敗

ボタン:

- `通知をコピー`
  現在の通知文をそのままクリップボードへコピーする。

### 3.2 セットアップカード

セットアップは完全JSONの入口である。  
ここにある操作は、すべて完全JSON欄を中心に動く。

置かれているもの:

- AI 用 prompt 種別 select
- `AI用プロンプトをコピー`
- 完全JSON欄 textarea
- 完全JSON欄直下の入出力ボタン
- サンプル select
- 実行系ボタン

完全JSON欄直下のボタン:

- `アップロード`
  JSON ファイルを読み込み、その文字列を完全JSON欄へ入れるだけ。自動で開始や再構築はしない。
- `ダウンロード`
  完全JSON欄に現在入っている文字列を、そのまま `.json` として保存する。
- `クリップボードへコピー`
  完全JSON欄の文字列をそのままコピーする。
- `クリップボードからペースト`
  クリップボードの文字列を完全JSON欄へ貼る。自動 validate はしない。

実行系ボタン:

- `サンプル読込`
  サンプル JSON を完全JSON欄へ入れる。再構築可能なら editor draft も更新される。
- `このJSONで開始`
  完全JSON欄を parse し、normalize し、validate し、問題がなければ実行 state を初期化して開始する。
- `最初からやり直す`
  直前に正常ロードされた完全JSONを使って、現在状態だけを初期化し直す。
- `現在状態をコピー`
  実行中 state と log を JSON としてコピーする。続篇生成やデバッグ向け。
- `続篇用プロンプトをコピー`
  現在の state を引き継ぐ AI 用 prompt をコピーする。

### 3.3 エディタカード

エディタは次の 2 モードを持つ。

- `JSON直接編集モード`
- `基本エディタモード`

`JSON直接編集モード` は完全JSON欄を人間が直接編集する前提のモードである。  
`基本エディタモード` はカテゴリ別フォームで編集する前提のモードである。

基本エディタモードでできること:

- カテゴリ切替
- 項目一覧から 1 件選択
- 新規作成
- 複製
- 削除
- フォーム編集
- `JSONへ反映`
- `JSONから再構築`

`JSONへ反映` の意味:

- 現在の editor draft を統合JSONへ組み立て直す
- その結果を完全JSON欄へ書き戻す
- この時点ではゲーム開始はしない

`JSONから再構築` の意味:

- 完全JSON欄を parse / normalize / validate する
- 問題がなければ editor draft を作り直す
- 基本エディタの各カテゴリは、その draft を元に描画される

### 3.4 AI支援カード

AI支援カードは基本エディタモードのときだけ表示される。  
このカードは完全JSON全体ではなく、現在カテゴリのやりとりに特化している。

置かれているもの:

- AI支援通知
- カテゴリJSON欄 textarea
- カテゴリJSON欄直下の入出力ボタン
- `現在カテゴリをカテゴリJSON欄へ反映`
- `プロンプト+カテゴリJSONをコピー`
- `カテゴリJSON欄の内容をマージ`

カテゴリJSON欄直下のボタン:

- `アップロード`
  JSON ファイルの文字列をカテゴリJSON欄へ入れるだけ。
- `ダウンロード`
  カテゴリJSON欄の文字列をそのまま保存する。
- `クリップボードへコピー`
  カテゴリJSON欄の文字列をそのままコピーする。
- `クリップボードからペースト`
  クリップボードの文字列をカテゴリJSON欄へ貼る。

AI支援系ボタン:

- `現在カテゴリをカテゴリJSON欄へ反映`
  現在の editor 選択内容からカテゴリJSONを生成し、カテゴリJSON欄へ入れる。
- `プロンプト+カテゴリJSONをコピー`
  選択中の prompt とカテゴリJSON欄の中身を一体化してコピーする。
  このとき prompt 内の `【見本JSON】` セクションは自動で除去される。
- `カテゴリJSON欄の内容をマージ`
  カテゴリJSON欄を parse し、editor draft にマージし、完全JSON欄にも反映する。

### 3.5 進行カード

現在の進行ヘッダを表示する。

表示内容:

- タイトル
- 現在イベント ID
- 現在スポット
- 時刻
- 既読数
- 体力

### 3.6 状態カード

現在状態の概要を表示する。  
必要なら raw JSON も開ける。

表示内容:

- 所持アイテム
- プレイヤー
- active scenario
- 主クエスト
- 進行中クエスト
- current_event_id
- current_spot_id
- current spot 上のアイテム
- NPC の位置と状態
- flags

ボタン:

- `詳細JSONを表示`
  現在 state の raw JSON を開閉する。

### 3.7 本文カード

現在 event の本文と選択肢を表示する。

表示内容:

- event title
- speaker line
- text
- conditional_texts を反映した追記本文
- actions / choices のボタン

### 3.8 ログカード

進行ログを時系列で表示する。  
action や choice による進行履歴、終了、失敗などが残る。

## 4. 完全JSONの正規形

現行実装の正規形は次である。

```json
{
  "title": "string",
  "version": "string",
  "definitions": {
    "characters": {},
    "items": {}
  },
  "world": {
    "areas": {},
    "dungeons": {},
    "spots": {}
  },
  "events": {},
  "scenarios": {},
  "quests": {},
  "initial_state": {}
}
```

top-level で重要な考え方は次のとおり。

- `definitions` は定義情報
- `world` は常設世界
- `events` は event 本体の保管庫
- `scenarios` は scenario 単位の参照と差分設定
- `quests` は quest 単位の参照と目標定義
- `initial_state` は開始時 state

### 4.1 `title`

ゲーム全体のタイトル。  
進行ヘッダ表示やダウンロードファイル名の候補に使う。

### 4.2 `version`

任意の管理文字列。  
現行 runtime は分岐には使わないが、JSON 自体の版管理には使える。

## 5. `definitions` の詳細

`definitions` は「辞書的な定義」であり、現在状態そのものではない。

### 5.1 `definitions.characters`

キャラクター定義を置く。

典型的な用途:

- 表示名の管理
- 役割タグの保持
- 初期説明文
- base stats の辞書

よく使うキー:

- `name`
  表示名。
- `tags`
  `player`、`npc`、`detective` のような自由タグ。
- `description`
  キャラの定義説明。
- `base_stats`
  設定上の基礎値。実行時の現在値とは別。

実行時にどう使われるか:

- `participants` の参照先として validate される
- 状態カードの説明補助になる
- 現在 state 側の character と ID で対応づく

例:

```json
{
  "holmes": {
    "name": "シャーロック・ホームズ",
    "tags": ["player", "detective"],
    "description": "観察と推理に優れる探偵。",
    "base_stats": {
      "体力": 60,
      "推理力": 8
    }
  }
}
```

### 5.2 `definitions.items`

アイテム定義を置く。

よく使うキー:

- `name`
  表示名。
- `item_type`
  `key`、`document`、`tool` などの分類。
- `tags`
  `quest`、`evidence`、`consumable` などの自由タグ。
- `description`
  アイテム説明。
- `stackable`
  複数所持を想定するか。
- `consumable`
  消耗品として扱うかのメタ情報。

実行時にどう使われるか:

- inventory 表示
- `required_items` の参照先
- `add_items`、`remove_items`、`remove_spot_items` の validate

例:

```json
{
  "brass_key": {
    "name": "真鍮の鍵",
    "item_type": "key",
    "tags": ["quest", "unlock"],
    "description": "古い金庫を開けるための鍵。",
    "stackable": false,
    "consumable": false
  }
}
```

## 6. `world` の詳細

`world` は scenario より前に存在する常設世界である。  
大きく見ると、`areas`、`dungeons`、`spots` の 3 層で地理を持つ。

### 6.1 `world.areas`

大きな地理単位を置く。

主なキー:

- `name`
- `description`

使いどころ:

- 地域説明
- spot や dungeon の親単位

### 6.2 `world.dungeons`

建物、施設、ダンジョン、まとまり空間を置く。

主なキー:

- `name`
- `area_id`
- `type`
- `description`

`area_id` の意味:

- この dungeon が属する area
- validate では `world.areas` に存在するかを見る

### 6.3 `world.spots`

プレイヤーや NPC が実際に立つ場所を置く。  
event の `spot_id` は基本的にここを参照する。

主なキー:

- `name`
- `area_id`
- `dungeon_id`
- `spot_type`
- `connections`
- `description`

`connections` の意味:

- 通常移動候補の基礎データ
- 移動系 UI で参照される
- target_spot の整合確認にも使われる

例:

```json
{
  "study": {
    "name": "書斎",
    "area_id": "sealed_manor",
    "dungeon_id": "moran_manor",
    "spot_type": "room",
    "connections": ["entrance_hall", "hidden_passage"],
    "description": "本棚に囲まれた書斎。"
  }
}
```

## 7. `events` の詳細

現行正規形では、event 本体はすべて top-level `events` に置く。  
scenario や quest は event 本体を持たず、参照だけを持つ。

### 7.1 event の最小イメージ

```json
{
  "study_event": {
    "title": "書斎",
    "event_type": "spot_event",
    "spot_id": "study",
    "participants": ["holmes", "watson"],
    "text": [
      "本棚に囲まれた書斎だ。",
      "机には書類が散らばっている。"
    ],
    "conditional_texts": [],
    "actions": [],
    "choices": []
  }
}
```

### 7.2 event の主要キー

`title`

- event の表示タイトル。

`event_type`

- 文字列で自由に置ける分類。
- 現行サンプルでは `spot_event`、`ending`、`game_over` などを使う。
- ending / game_over の専用 UI は、ここだけでなく event 本体の内容と遷移文脈も踏まえる。

`spot_id`

- その event が属する spot。
- 現在 spot と一致したとき、可視候補になりうる。

`participants`

- 登場人物 ID 配列。
- validate では `initial_state.characters` への存在を確認する。

`speaker`

- 行頭表示に使う発話者名。任意。

`text`

- 文字列または文字列配列。
- 配列なら行ごとに表示される。
- ここが本文の基礎で、`conditional_texts` はこれに追加される。

`conditional_texts`

- 条件を満たしたときだけ追加で表示する本文群。
- 再訪時の描写変化や、NPC 配置に応じた差分に使う。

`actions`

- その場処理用の選択肢。
- アイテム取得、仕掛け操作、会話、調査などに向く。

`choices`

- 遷移を主目的とする選択肢。
- ただし effects を持てるので、単なる移動以上のこともできる。

### 7.3 `conditional_texts` の形

```json
{
  "conditions": {
    "required_flags": ["cipher_decoded"]
  },
  "text": [
    "暗号はすでに解読済みだ。"
  ]
}
```

使いどころ:

- 一度調べた後だけ説明を変える
- 特定の item 所持時だけ補足を出す
- NPC がその場にいるときだけ描写を足す
- quest の進行後だけ場所の意味を変える

### 7.4 `actions` / `choices` で意味を持つ主なキー

`text`

- ボタン文言。

`conditions`

- 表示可能か、実行可能かを決める条件。

`effects`

- 実行時に state をどう更新するか。

`next_event_id`

- 成功後に直接飛ぶ event ID。

`target_spot_id`

- 主に移動先の説明用。
- 実行時には移動候補の解釈にも使われる。

`disabled_text`

- 条件未達時の代替ボタン文言。

`failure_message`

- 条件未達時の通知文。

`success_message`

- 成功時の通知文。

`hidden_when_disabled`

- 条件未達時、ボタンごと非表示にするかどうか。

### 7.5 `next_event_id` の解釈

- 文字列なら、その event へ直接遷移する。
- `null` なら event 終端として扱われる。
- 省略された場合でも、effects により spot が変わっていれば、その spot の可視 event から次が解決される。

### 7.6 `actions` と `choices` の実務的な使い分け

現行実装は両方とも内部的には似た option として処理する。  
ただし、作者側の運用としては次の使い分けを推奨する。

- `actions`
  その場で調べる、拾う、押す、話す、開けるなど
- `choices`
  別 event へ進む、別 spot へ移る、結論を選ぶなど

AI に生成させるときも、この使い分けを守った方が読める JSON になる。

## 8. `scenarios` の詳細

scenario は event 本体ではなく、進行単位、主線単位、世界差分単位として扱う。

主なキー:

- `title`
- `description`
- `start_event_id`
- `game_over_event_id`
- `event_ids`
- `movement_rules`
- `overlay`

各キーの意味:

`title`

- scenario の表示名。

`description`

- scenario の説明文。

`start_event_id`

- その scenario の開始に最もふさわしい event。
- 初期 current_event が空の場合、ここが優先候補になる。

`game_over_event_id`

- game over に飛ばす代表 event。
- 体力低下などの文脈でここへ飛ぶ処理がある。

`event_ids`

- この scenario 専用とみなす event ID 群。
- 脱出本線や大筋の進行イベントをここに置く。

`movement_rules`

- scenario 固有の移動制御用設定置き場。
- 現行実装では editor で編集対象になっており、scenario 設計上の補助情報として保持できる。
- 一方で、現在の runtime は `movement_rules` を強く解釈して挙動を切り替える中心機能にはしていない。
- そのため、ここは「将来拡張を見据えた scenario 側設定欄」に近い。

`overlay`

- active scenario の間だけ world / definitions に差分を足すためのキー。
- 現行実装の「シナリオ主導型」要素の中心である。

運用上の考え方:

- scenario は「大きい章」や「主線」を持つ単位
- quest より広く、世界の主目的を表す
- scenario 切替は `effects.start_scenario` で行う

### 8.1 `scenario.overlay` の意味

`overlay` は、scenario が有効な間だけ世界や定義へ差分を適用する仕組みである。  
完全JSONの base world を直接書き換えるのではなく、「その scenario ではこう見える」を作る。

現行実装が runtime / validate の両方で解釈しているキー:

- `add_areas`
- `add_dungeons`
- `add_spots`
- `disabled_spot_ids`
- `add_characters`
- `add_items`

### 8.1.0 `scenario.overlay` の基本的な考え方

`overlay` は「scenario 開始後にだけ見える差分辞書」である。  
base 側にある `world` や `definitions` を消して別物に置き換えるのではなく、active scenario に応じて上から足す。

内部で起きていること:

- area 一覧を取得するときは `world.areas` を複製し、その上に `overlay.add_areas` を重ねる
- dungeon 一覧も同様に `overlay.add_dungeons` を重ねる
- spot 一覧は `overlay.add_spots` を重ねた後、`disabled_spot_ids` を削除する
- character 定義は `definitions.characters` に `overlay.add_characters` を重ねる
- item 定義は `definitions.items` に `overlay.add_items` を重ねる

つまり `overlay` は state ではなく「定義レイヤー」に効く。  
現在状態の flags や inventory を直接変える機能ではない。

### 8.1.0.1 何に向いているか

`overlay` は次の用途に向いている。

- scenario 開始後にだけ行ける新エリアを出す
- scenario 中だけ存在する部屋や通路を作る
- scenario 中だけ既存 spot を閉鎖する
- scenario 中だけ有効な NPC 定義を足す
- scenario 中だけ使う item 定義を足す

逆に向いていないもの:

- プレイヤー inventory の直接変更
- flags の直接変更
- 体力の直接変更
- あるキャラを特定 spot に瞬時移動させること

それらは `effects` や `initial_state` 側で行う。

### 8.1.0.2 どうやって有効になるか

`overlay` は、その scenario が active になった時だけ有効になる。

有効化の方法:

- `initial_state.active_scenario_id` で最初から有効にする
- event の `effects.start_scenario` で途中から切り替える

つまり overlay を書いただけでは効かない。  
その scenario が active である必要がある。

### 8.1.0.3 使用手順

基本的な使い方は次の順で考える。

1. base world に常設の `areas / dungeons / spots / characters / items` を置く
2. 特定 scenario でだけ増えるものを `overlay.add_*` に置く
3. 特定 scenario でだけ消したい base spot を `disabled_spot_ids` に置く
4. 開始時または event の effects で `active_scenario_id` をその scenario にする
5. その scenario の `event_ids` や `start_event_id` と合わせて進行を構成する

### 8.1.0.4 よくある実践パターン

パターン A: 後半章で新しい区画を開放する

- 前半では base world のみ見せる
- `start_scenario` で後半 scenario に切り替える
- 後半 scenario の `overlay.add_spots` に新しい区画を定義する

パターン B: 災害や封鎖で一部の道を潰す

- base world に通常の spot 群を置く
- 緊急 scenario の `disabled_spot_ids` に閉鎖 spot を入れる

パターン C: scenario 限定 NPC を出す

- base `definitions.characters` には置かない
- 対象 scenario の `overlay.add_characters` にだけ入れる
- 必要なら `initial_state.characters` に状態を持たせるか、event effects で `spot_id` を与える

パターン D: scenario 限定アイテムを導入する

- `overlay.add_items` で item 定義を足す
- その item を `spot_states` や `add_items` で実際に取得可能にする

### 8.1.0.5 使用時の重要な注意

`overlay` は「定義を足す」機能であり、「状態を生む」機能ではない。

たとえば `add_characters` だけ書いても:

- そのキャラの定義は見える
- しかし `initial_state.characters` に状態がなければ、現在位置や current_stats は持たない
- event の `participants` 参照には使えても、位置条件などでは不足する可能性がある

たとえば `add_items` だけ書いても:

- item 定義は見える
- しかし inventory や `spot_states` に置かなければ、実際には取得できない

たとえば `add_spots` だけ書いても:

- spot 定義は見える
- しかし event がなければ本文は出ない
- `connections` や移動 choice がなければ行き来しにくい

つまり overlay を使うときは、次もセットで考える必要がある。

- event をどこに置くか
- その spot にどう到達するか
- NPC や item の現在状態をどう与えるか

### 8.1.0.6 `overlay` と `event_ids` の関係

`overlay` は世界差分、`event_ids` は event 所属であり、役割が違う。

- `overlay`
  世界や定義を増減させる
- `event_ids`
  その scenario で見える主線 event を指定する

実務上は両方を組み合わせる。

例:

- `overlay.add_spots` で新区域を作る
- `event_ids` にその区域の event を入れる
- `start_event_id` でその scenario の入口 event を指定する

### 8.1.0.7 `overlay` は quest ではなく scenario 側に置く

現行実装で `overlay` を持つのは scenario だけである。  
quest は event 所属や目標管理を担うが、world 差分は持たない。

そのため:

- 「世界の見え方を変える」は scenario
- 「調査目的を足す」は quest

と分けて考えるのが正しい。

### 8.1.0.8 最小例

```json
{
  "scenarios": {
    "mine_emergency": {
      "title": "鉱区非常事態",
      "start_event_id": "mine_gate_event",
      "event_ids": ["mine_gate_event", "deep_tunnel_event"],
      "overlay": {
        "add_spots": {
          "deep_tunnel": {
            "name": "深部坑道",
            "area_id": "mine_area",
            "spot_type": "tunnel",
            "connections": ["mine_gate"],
            "description": "事故後にだけ立ち入り可能になった深部坑道。"
          }
        },
        "disabled_spot_ids": ["old_market"],
        "add_items": {
          "rescue_lamp": {
            "name": "救助用ランプ",
            "item_type": "tool",
            "description": "非常時にだけ配布される強力なランプ。"
          }
        }
      }
    }
  }
}
```

この例で起きること:

- `mine_emergency` が active なら `deep_tunnel` が world.spots に見える
- `old_market` は一時的に使えなくなる
- `rescue_lamp` の item 定義が参照可能になる
- ただし `rescue_lamp` を実際に持たせるには別途 `spot_states` または `effects.add_items` が必要

#### 8.1.1 `add_areas`

追加 area 定義。  
active scenario のときだけ `world.areas` に合成される。

例:

```json
{
  "overlay": {
    "add_areas": {
      "collapsed_mine": {
        "name": "崩落鉱区",
        "description": "本編後半でのみ進入できる危険区域。"
      }
    }
  }
}
```

#### 8.1.2 `add_dungeons`

追加 dungeon 定義。  
scenario 限定建物や区画を出すために使う。

#### 8.1.3 `add_spots`

追加 spot 定義。  
scenario 固有の部屋、イベント用の臨時地点、後半だけ開く区画などに使う。

これは実際の runtime で可視 spot 候補や validate に反映される。  
つまり「scenario 中だけ存在する spot」を本当に作れる。

#### 8.1.4 `disabled_spot_ids`

既存 spot を scenario 中だけ無効化する。

向いている用途:

- 崩落で通れなくなる
- 閉鎖で立ち入り不能になる
- 別章では訪問できない区画にする

#### 8.1.5 `add_characters`

これが旧感覚でいう `add_character` に相当する。  
ただし現行仕様では単数キーではなく、`overlay.add_characters` というコレクション追加で扱う。

意味:

- active scenario の間だけ `definitions.characters` にキャラ定義を追加する
- validate でもその scenario が active なら参照可能キャラとして認識される

例:

```json
{
  "overlay": {
    "add_characters": {
      "mine_guard": {
        "name": "鉱区の監視員",
        "tags": ["npc", "guard"],
        "description": "崩落区域を監視している臨時警備員。"
      }
    }
  }
}
```

重要な注意:

- これは「定義追加」であり、現在 state への自動出現ではない
- 実際にそのキャラをその場へ存在させたいなら、`initial_state.characters` または `effects.set.characters.<id>.spot_id` 側も合わせて設計する必要がある

#### 8.1.6 `add_items`

active scenario の間だけ `definitions.items` に item 定義を追加する。

向いている用途:

- scenario 固有アイテム
- 後半章だけ使う鍵や証拠
- 本編では存在しない専用ツール

### 8.2 シナリオ主導型とは何か

現行実装でいう「シナリオ主導型」は、単に `active_scenario_id` を持つだけではなく、次を scenario が主導する構造を指す。

- どの event 群が主線として見えるか
- 開始 event は何か
- game over の代表 event は何か
- その scenario 中だけ有効な world 差分は何か
- その scenario 中だけ有効な character / item 定義は何か

つまり scenario は「event の所属先」であるだけでなく、「今の章で見えている世界そのもの」を差分的に決める。

## 9. `quests` の詳細

quest は個別目標や副線を表す。  
event 本体ではなく参照だけを持つ。

主なキー:

- `title`
- `description`
- `start_event_id`
- `event_ids`

各キーの意味:

`title`

- quest 名。

`description`

- quest の意図やゴール。

`start_event_id`

- quest 開始時に見せたい event。

`event_ids`

- その quest に属する event 群。
- 事件解明、証拠集め、個別目標などをここに置く。

運用上の考え方:

- `effects.start_quest` で進行中 quest に加える
- `effects.set_primary_quest` で主 quest として前面に出す
- `effects.complete_quest` で完了処理する

## 10. event 所属の考え方

現行実装では event に 3 種類の所属概念がある。

### 10.1 scenario 専用 event

`scenario.event_ids` に入っている event。

向いている用途:

- 脱出本線
- scenario 固有の危機
- scenario 固有エンディング

### 10.2 quest 専用 event

`quest.event_ids` に入っている event。

向いている用途:

- 調査
- 推理
- 証拠回収
- 個別目標達成

### 10.3 未所属 event

どの scenario / quest の `event_ids` にも入っていない event。

向いている用途:

- 雰囲気イベント
- 寄り道イベント
- 世界観補足
- どうでもよいが存在感を作る小イベント

### 10.4 実行時に event 候補へ入るもの

現在可視候補になるのは次の和集合である。

- active scenario の `start_event_id`
- active scenario の `game_over_event_id`
- active scenario の `event_ids`
- active quest 全件の `start_event_id`
- active quest 全件の `event_ids`
- 未所属 event

つまり:

- 主線は scenario で出る
- 進行中 quest のイベントも出る
- 未所属 event は常時補助候補として残る

## 11. `initial_state` の詳細

`initial_state` は開始時 state の雛形であり、実行中 state の元になる。

主なキー:

- `player_character_id`
- `active_scenario_id`
- `active_quest_ids`
- `primary_quest_id`
- `current_quest_id`
- `current_event_id`
- `current_spot_id`
- `time`
- `flags`
- `unique`
- `characters`
- `spot_states`
- `quest_states`

### 11.1 `player_character_id`

- プレイヤーとして扱う character ID。
- これを基準に inventory や現在位置の主要表示が決まる。

### 11.2 `active_scenario_id`

- 現在有効な scenario。
- visible event 解決の基準になる。

### 11.3 `active_quest_ids`

- 現在進行中の quest 一覧。
- visible event 解決で quest 側 event を拾うために使う。

### 11.4 `primary_quest_id`

- UI 上の主クエスト。
- `current_quest_id` よりこちらが主扱いである。

### 11.5 `current_quest_id`

- 旧互換寄りの現在 quest。
- normalize や state 補完時に `primary_quest_id` に寄せられる。

### 11.6 `current_event_id`

- 現在 event を直接指定する優先キー。
- ここが有効なら、開始時はまずこの event を見にいく。

### 11.7 `current_spot_id`

- 現在 spot の互換キー。
- 実装上は `characters[player_character_id].spot_id` と同期される。

### 11.8 `time`

- 現在時刻。
- `year`、`month`、`day`、`hour`、`minute`、`turn` を持てる。
- `effects.add_time` で進められる。

### 11.9 `flags`

- 文字列配列。
- 一度成立した事実や解除済み状態を保持する。

### 11.10 `unique`

- 任意の数値辞書。
- 推理力、信用値、恐怖値など、作品固有の抽象値に向く。

### 11.11 `characters`

- 各キャラクターの現在状態。
- `definitions.characters` が定義、こちらが実行状態。

1 キャラの典型例:

```json
{
  "holmes": {
    "spot_id": "entrance_hall",
    "status": "active",
    "current_stats": {
      "体力": 60
    },
    "inventory": []
  }
}
```

よく使うキー:

- `spot_id`
  現在位置。
- `status`
  任意文字列の状態。
- `current_stats`
  現在能力値。体力など。
- `inventory`
  所持アイテム一覧。
- `relationships`
  対人関係の補助辞書。

### 11.12 `spot_states`

- spot ごとの現在状態を持つ。
- 今は主に、その場に置かれているアイテム管理に使う。

例:

```json
{
  "study": {
    "items": [
      {
        "item_id": "cipher_note",
        "quantity": 1,
        "container_id": "desk_papers"
      }
    ]
  }
}
```

### 11.13 `quest_states`

- quest ごとの現在状態を持つ。
- `start_quest` や `complete_quest` により補完・更新される。

1 件の典型形:

```json
{
  "solve_case": {
    "status": "active",
    "progress_flags": [],
    "completed_at": null
  }
}
```

## 12. normalize がしていること

正規形は top-level `events` 方式だが、実装では旧形式を最小限吸収する。  
ただし、今後 AI や人間が新規作成するなら、この文書の正規形を直接使うべきである。

normalize が行う主な処理:

- `scenario` 単体しかない入力を `scenarios` へ持ち上げる
- `scenario.quests` を top-level `quests` へ上げる
- `scenario.events` を top-level `events` へ上げる
- `world.base_events` を top-level `events` へ上げる
- `event_ids` が未設定なら、旧 event 群から補完する
- `current_quest_id` から `active_quest_ids` と `primary_quest_id` を補完する
- `primary_quest_id` が active に入っていなければ補う
- player spot と `current_spot_id` の整合を取る
- 各 collection がオブジェクトであることを最低限保証する

重要な注意:

- すでに `scenario.event_ids` があるなら、旧 event 持ち上げ後もその明示所属を優先する
- つまり event 所属を作者が明示したなら、それを壊さない

## 13. 条件判定 `conditions` の詳細

条件判定は `checkConditions()` でまとめて評価される。  
評価順は次のとおり。

1. flags 条件
2. NPC 位置条件
3. inventory 条件
4. state 数値条件
5. unique 数値条件

いずれか 1 つでも不成立なら false になる。

### 13.1 flags 条件

主なキー:

- `required_flags`
- `forbidden_flags`

例:

```json
{
  "required_flags": ["cipher_decoded"],
  "forbidden_flags": ["hidden_passage_open"]
}
```

意味:

- `cipher_decoded` が立っている必要がある
- `hidden_passage_open` が立っていたら失敗する

### 13.2 NPC 位置条件

主なキー:

- `required_character_spots`
- `forbidden_character_spots`

例:

```json
{
  "required_character_spots": {
    "watson": ["study", "entrance_hall"]
  }
}
```

意味:

- `watson` が `study` または `entrance_hall` にいれば成立する

実務的な使い道:

- NPC がその場にいないと手伝えない
- NPC が危険地帯にいるときだけ特殊文を出す
- 別行動中かどうかを spot で表す

### 13.3 inventory 条件

主なキー:

- `required_items`

受理される形:

- `"item_id"` という単純文字列
- `{ "item_id": "brass_key", "quantity": 1 }`

例:

```json
{
  "required_items": [
    { "item_id": "brass_key", "quantity": 1 }
  ]
}
```

意味:

- プレイヤー inventory に `brass_key` が 1 個以上必要

### 13.4 state 数値条件

state 上の path に対して数値比較できる。  
体力や任意の現在値を判定するための仕組みである。

主なイメージ:

- `min_state`
- `max_state`

例として使いやすい path:

- `characters.holmes.current_stats.体力`
- `quest_states.solve_case.progress`

### 13.5 unique 数値条件

`state.unique` 内の数値を比較できる。

主なイメージ:

- `min_unique`
- `max_unique`

向いている用途:

- 推理力が一定以上なら見抜ける
- 恐怖値が一定以上なら別描写を出す
- 信用値が高いと協力してもらえる

## 14. effects の詳細

effects は action / choice 実行時の state 更新である。  
`applyEffects()` 内では次の順で処理される。

1. `add`
2. `set`
3. flags 操作
4. `add_unique`
5. inventory 操作
6. spot 上アイテム操作
7. 時間操作
8. scenario / quest 操作

この順序があるため、同じ action の中で `set` 済み state を後段処理が参照することがある。

### 14.1 `add`

数値 path に加算する。

例:

```json
{
  "add": {
    "characters.holmes.current_stats.体力": -5
  }
}
```

意味:

- `characters.holmes.current_stats.体力` を 5 減らす

使いどころ:

- 体力減少
- 所要ターンの蓄積
- 任意数値の増減

### 14.2 `set`

path 指定で値を代入する。

例:

```json
{
  "set": {
    "current_spot_id": "study",
    "characters.holmes.spot_id": "study",
    "characters.watson.spot_id": "basement"
  }
}
```

意味:

- プレイヤーの現在 spot を `study` にする
- プレイヤー本体の `spot_id` を `study` にする
- Watson を `basement` に移動する

`set` の重要な使い道:

- spot 移動
- NPC 再配置
- 任意 state 値の直接切替
- 今いる場所の切替を choice の effects だけで表現する

### 14.3 `set_flags`

flags を追加する。

例:

```json
{
  "set_flags": ["cipher_decoded", "vault_opened"]
}
```

意味:

- 配列内の flag を state.flags へ追加する
- 重複は実質吸収される

### 14.4 `remove_flags`

flags を削除する。

例:

```json
{
  "remove_flags": ["temporary_hint"]
}
```

向いている用途:

- 一時ヒント解除
- 一度きりの誘導文抑止
- 緊急状態の解除

### 14.5 `add_unique`

`state.unique` 内の数値を加減算する。

例:

```json
{
  "add_unique": {
    "推理力": 1,
    "恐怖値": -2
  }
}
```

### 14.6 `add_items`

プレイヤー inventory に item を加える。

受理される形:

- `"item_id"`
- `{ "item_id": "brass_key", "quantity": 1 }`

例:

```json
{
  "add_items": [
    { "item_id": "brass_key", "quantity": 1 }
  ]
}
```

意味:

- 指定 item を inventory に加える
- 文字列だけなら数量 1 として扱われる

### 14.7 `remove_items`

プレイヤー inventory から item を減らす。

受理される形:

- `"item_id"`
- `{ "item_id": "iron_bar", "quantity": 1 }`

意味:

- 使用済みアイテムの消費
- 支払い、破損、提出など

### 14.8 `remove_spot_items`

spot_states 側から item を除去する。

例:

```json
{
  "remove_spot_items": [
    { "spot_id": "study", "item_id": "cipher_note", "quantity": 1 }
  ]
}
```

意味:

- `study` に置かれている `cipher_note` を 1 個消す

`spot_id` を省略した場合:

- 現在 spot から取り除く

典型用途:

- 「拾う」を実行したあと、その場から消す
- 一度回収した証拠を再取得できなくする

### 14.9 `add_time`

時間を進める。

例:

```json
{
  "add_time": {
    "minutes": 5
  }
}
```

意味:

- `time.minute` を進める
- くり上がり処理も行う
- `turn` も進む

### 14.10 `start_scenario`

active scenario を切り替える。

例:

```json
{
  "start_scenario": "escape_manor_scenario"
}
```

意味:

- `active_scenario_id` を更新する
- `current_event_id` を空にする
- 次の event 解決は新 scenario の参照を基準に行う

### 14.11 `start_quest`

quest を開始する。

例:

```json
{
  "start_quest": "solve_case"
}
```

意味:

- `active_quest_ids` に追加する
- `quest_states.solve_case` がなければ作る
- 初期状態は `status: "active"`、`progress_flags: []`、`completed_at: null`

### 14.12 `set_primary_quest`

主 quest を切り替える。

例:

```json
{
  "set_primary_quest": "solve_case"
}
```

意味:

- `primary_quest_id = "solve_case"`
- `current_quest_id = "solve_case"`
- 必要なら active quest 側にも補う

### 14.13 `complete_quest`

quest を完了状態にする。

例:

```json
{
  "complete_quest": "solve_case"
}
```

意味:

- `quest_states.solve_case.status = "completed"`
- `quest_states.solve_case.completed_at = 現在 time の複製`

## 15. option の移動先解釈

option の移動先は次の優先順で解釈される。

1. `option.target_spot_id`
2. `effects.set.characters.<player>.spot_id`
3. `effects.set.current_spot_id`

つまり:

- 移動 choice を作るとき、`target_spot_id` がなくても移動は成立しうる
- ただし可読性のため、移動 choice には `target_spot_id` を併記した方がよい

## 16. event 進行の流れ

現在 event が描画されるまでの流れは概ね次のとおり。

1. `current_event_id` が有効なら、その event を優先する
2. なければ現在 spot と可視 event 候補から最初の visible event を探す
3. event の `text` を表示する
4. `conditional_texts` を条件評価して追加する
5. `actions` と `choices` をボタン化する
6. ボタン押下時に `conditions` を再評価する
7. 成立していれば `effects` を適用する
8. `next_event_id` があればそこへ遷移する
9. なければ現在 spot と可視 event 候補から次を解決する

実務的な注意:

- event の見え方は `spot_id` と所属だけでなく conditions にも左右される
- 同じ spot に複数 event を置いても、条件で順次出し分けられる

## 17. ending / game_over の扱い

現在 event が ending や game_over の文脈になった場合、通常の option UI ではなく終了系 UI を出す。

ユーザーができること:

- `最初からやり直す`
- `続篇用プロンプトをコピー`

また、体力低下などの条件で `game_over_event_id` へリダイレクトする処理がある。  
そのため、scenario を作るときは `game_over_event_id` を用意しておくと扱いやすい。

## 19. カテゴリJSONの正確な受理形式

カテゴリJSONは「完全JSONの一部だけを安全にやり取りする形式」である。  
`classifyJsonPayload()` が受理するのは、概ね次のキー群だけである。

受理されるカテゴリJSON top-level キー:

- `areas`
- `dungeons`
- `spots`
- `characters`
- `items`
- `events`
- `scenarios`
- `quests`
- `scenario_id`
- `scenario`
- `quest_id`
- `quest`
- `initial_state_fragment`

### 19.1 collection 単位のカテゴリJSON

例:

```json
{
  "areas": {
    "hamakita_station_front": {
      "name": "浜北駅前",
      "description": "駅前エリア。"
    }
  }
}
```

意味:

- `world.areas` の一部または全部としてマージする

### 19.2 scenario 単体差分

例:

```json
{
  "scenario_id": "escape_manor_scenario",
  "scenario": {
    "title": "モラン邸からの脱出",
    "event_ids": ["entrance_hall_event"]
  }
}
```

意味:

- `scenarios.escape_manor_scenario` に対する単体差分

### 19.3 quest 単体差分

例:

```json
{
  "quest_id": "solve_case",
  "quest": {
    "title": "事件の真相を暴く",
    "event_ids": ["study_event"]
  }
}
```

### 19.4 initial_state 差分

例:

```json
{
  "initial_state_fragment": {
    "active_scenario_id": "escape_manor_scenario",
    "flags": ["demo_started"]
  }
}
```

意味:

- `initial_state` 全体ではなく、一部だけを deep merge する

## 20. カテゴリJSONマージの規則

カテゴリJSONマージは `mergeFragmentIntoDraft()` と `deepMergeObjects()` で行われる。

規則は次のとおり。

### 20.1 オブジェクトは deep merge

- 既存 object がある場合、同じ key は上書き
- 別 key は残す

### 20.2 配列は置換

- 配列同士は要素単位マージではない
- patch 側の配列で丸ごと置き換える

そのため:

- `event_ids`
- `participants`
- `actions`
- `choices`
- `flags`

などを AI に返させるときは、「追加分だけ」ではなく「その項目の完成形」を返させる方が安全である。

### 20.3 存在しない ID は追加

- 既存 collection にない ID が来たら新規追加される

### 20.4 マージ後は normalize と validate が走る

- editor draft に反映
- normalize で整形
- 完全JSON欄にも統合反映
- 破綻していれば失敗扱いになる

## 21. AI用 prompt の種類

現行 prompt select では、完全JSON用とカテゴリJSON用の両方を扱う。

### 21.1 `generic`

完全JSON 1 本を生成させる総合 prompt。  
見本JSONを含む。

### 21.2 `shindo`

ホラー寄り完全JSON用 prompt。  
見本JSONを含む。

### 21.3 `editor_area` などの `editor_*`

カテゴリJSONのみを返させる prompt。

代表例:

- `editor_area`
  `{"areas": {...}}`
- `editor_event`
  `{"events": {...}}`
- `editor_scenario`
  `{"scenarios": {...}}` または scenario 単体差分
- `editor_quest`
  `{"quests": {...}}` または quest 単体差分

### 21.4 `editor_merge_assistant`

完全JSONではなくカテゴリJSONだけを返すよう誘導する相談用 prompt。

### 21.5 `AI用プロンプトをコピー`

- prompt 本文だけをコピーする
- 見本JSONも含む

### 21.6 `プロンプト+カテゴリJSONをコピー`

- prompt 本文とカテゴリJSON欄の内容をまとめてコピーする
- prompt の `【見本JSON】` セクションは除去される
- 代わりに `【現在のカテゴリJSON】` セクションとして今のカテゴリ内容を渡す

## 22. `続篇用プロンプトをコピー` の意味

この機能は、現在の state を引き継いだ続篇や追加章を AI に作らせるための補助である。

prompt 内で重視している情報:

- `active_scenario_id`
- `active_quest_ids`
- `primary_quest_id`
- `quest_states`
- top-level `events` を使うこと
- scenario / quest は event_ids や start_event_id で参照すること

つまり、続篇生成でも現行の top-level `events` 構造を維持する前提になっている。

## 23. validate が見ていること

validate はかなり広く整合を見る。  
単に JSON 構文が正しいだけでは通らない。

### 23.1 top-level

- `title` が空でない文字列か
- `definitions` が object か
- `world` が object か
- `events`、`scenarios`、`quests` が object か

### 23.2 world

- area / dungeon / spot が object か
- `spot.area_id` が存在する area を指すか
- `spot.dungeon_id` が存在する dungeon を指すか
- `connections` が spot 参照として妥当か

### 23.3 initial_state

- `player_character_id` が存在するか
- `active_scenario_id` が存在する scenario を指すか
- `primary_quest_id` が存在する quest を指すか
- `characters` が object か
- `spot_states` が object か

### 23.4 events

- `title` が適切か
- `text` が文字列または文字列配列か
- `spot_id` が存在する spot を指すか
- `participants` の character が存在するか
- `conditional_texts` が object 群として妥当か
- action / choice の構造が妥当か
- `next_event_id` の参照先が存在するか

### 23.5 conditions

- `required_character_spots`
- `forbidden_character_spots`
- `required_items`

などの構造と参照先。

### 23.6 effects

とくに次を見ている。

- `set` で `.spot_id` を書いているなら、その spot が存在するか
- `add_items` の item が存在するか
- `remove_items` の item が存在するか
- `remove_spot_items` の item / spot が存在するか
- `start_scenario` の参照先 scenario が存在するか
- `start_quest` の参照先 quest が存在するか
- `set_primary_quest` の参照先 quest が存在するか
- `complete_quest` の参照先 quest が存在するか

### 23.7 開始可能性

開始時に event を解決できるかも見る。

具体的には次のいずれかが必要になる。

- `initial_state.current_event_id`
- active scenario の `start_event_id`
- active scenario の `event_ids` にある開始地点対応 event

## 25. AI に渡すときの実務ルール

このツールは AI と往復できるが、AI に自由に完全JSONを書かせるより、カテゴリJSONで刻んだ方が安全である。

推奨手順:

1. 基本エディタで対象カテゴリを選ぶ
2. `現在カテゴリをカテゴリJSON欄へ反映`
3. `プロンプト+カテゴリJSONをコピー`
4. AI に差分作成を依頼する
5. 返ってきた JSON をカテゴリJSON欄へ貼る
6. `カテゴリJSON欄の内容をマージ`
7. 必要なら `JSONへ反映`
8. `このJSONで開始`

AI に対する注意:

- event 本体は top-level `events` に置かせる
- scenario / quest には event 本体を置かせない
- 配列は置換なので、既存配列の完成形を返させる
- `set` の path は実在 path を使わせる
- `remove_spot_items` には `spot_id` を明示させる方が安全
