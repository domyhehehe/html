# 改修依頼: 汎用サウンドノベルJSONランナーを「基本エディタ + AI断片生成/マージ対応ツール」に拡張する

## 0. 目的

既存の「汎用サウンドノベルJSONランナー 完全版」を、単なる全文JSON入力型ランナーから、以下の機能を持つ「基本エディタ付きオーサリングツール」へ段階的に拡張したい。

今回の主目的は3つある。

1. 巨大な統合JSONを直接手書きする負担を減らし、カテゴリ別に編集できる基本エディタを追加する
2. AIにカテゴリ単位のJSON断片を作らせるための専用プロンプトを用意する
3. AIが作った断片JSONを、必ずメッセージ欄（既存のJSON入力欄）に貼り付けた上で、内部データへ安全にマージできるようにする

このツールは、最終的な実行データとしては1つの統合JSONを扱うが、制作工程では以下のような分割運用を可能にしたい。

- world の一部だけAIに作らせる
- items だけAIに作らせる
- characters だけAIに作らせる
- scenario だけAIに作らせる
- quest だけAIに作らせる
- それらを後からマージし、1つの統合JSONへ組み上げる

## 1. 前提認識

現行実装は以下の特徴を持つ。

- top-level 構造は `title / version / definitions / world / scenarios / quests / initial_state`
- `definitions.characters` と `definitions.items` を持つ
- `world` は `areas / dungeons / spots / base_events` を持つ
- `scenario` は `overlay / movement_rules / event_rules / events` を持てる
- active scenario は `initial_state.active_scenario_id` で決まる
- 既存UIは JSON全文入力 + サンプル読込 + 実行 + 状態確認が中心
- 既存で AIプロンプトコピー機能（generic / shindo）は実装済み
- 既存で validateScript(), normalizeScriptStructure(), applyEffects() などの基盤がある

この既存構造を壊さず、後方互換をなるべく維持すること。

## 2. 今回の実装方針

今回は「完成版の高機能ノーコードエディタ」は作らない。
まずは以下の思想で実装する。

- 基本エディタを追加する
- 既存の JSON入力欄は残す
- AI出力は自動で内部へ流し込まない
- 必ずユーザーがメッセージ欄（既存のJSON入力欄）へ貼り付けたものだけを取り込む
- 取り込む際は、完全JSONか断片JSONかを判定して、安全にマージする
- 編集結果は最終的に統合JSONとして textarea に反映できる
- その統合JSONをそのまま既存ランナーで実行できる

## 7. AI支援機能の仕様

### 7.1 新規追加するAIプロンプト種別

既存の `generic` / `shindo` に加えて、以下を追加する。

- editor_area
- editor_dungeon
- editor_spot
- editor_character
- editor_item
- editor_scenario
- editor_quest
- editor_merge_assistant（任意）

### 7.2 プロンプトの役割

各プロンプトは「完全JSON」ではなく、基本的にはカテゴリ断片を出力させる。

例:

- area 用なら `{"areas": {...}}`
- character 用なら `{"characters": {...}}`
- scenario 用なら `{"scenario_id": "...", "scenario": {...}}` もしくは `{"scenarios": {...}}`
- quest 用なら `{"quests": {...}}`

出力形式は「純粋なJSONのみ」に限定すること。

### 7.3 重要要件

AI出力は絶対に自動取り込みしないこと。
ユーザーが必ずメッセージ欄（既存 textarea = scriptInput）に貼り付けることを前提にする。

その上で、

- 「この内容を断片としてマージ」
- 「この内容を完全JSONとして再読込」
  をユーザーが明示的に選べるようにする。

## 8. JSONマージ機能の仕様

### 8.1 取り込み元

取り込み元は2つ

1. メッセージ欄（scriptInput）に貼られたJSON
2. 外部ファイルから読み込んだJSON

### 8.2 判定対象

取り込むJSONは以下のいずれかとする。

#### A. 完全JSON

top-level に以下のいずれかを持つもの

- definitions
- world
- scenarios
- quests
- initial_state

これは既存の完全スクリプトとして扱える。

#### B. 断片JSON

以下のような部分構造のみを持つもの

- areas
- dungeons
- spots
- characters
- items
- scenarios
- quests
- scenario + scenario_id
- quest + quest_id
- initial_state_fragment

### 8.3 マージルール

断片JSONは、対応する内部オブジェクトへ shallow ではなく基本的に id単位で merge する。

例:

- `areas` -> `world.areas`
- `dungeons` -> `world.dungeons`
- `spots` -> `world.spots`
- `characters` -> `definitions.characters`
- `items` -> `definitions.items`
- `scenarios` -> `scenarios`
- `quests` -> `quests`

もし `scenario_id + scenario` 形式なら

- `scenarios[scenario_id] = merge(existing, scenario)`

もし `quest_id + quest` 形式なら

- `quests[quest_id] = merge(existing, quest)`

### 8.4 競合時の基本動作

競合時はまず「上書きマージ」でよい。
ただし安全のために:

- 対象ID
- 既存の有無
- 上書き件数
  を通知欄に出すこと。

可能なら将来的に

- 上書き
- スキップ
- 比較
  を増やせる構造にしておく。

## 9. 外部保存 / 読み込み

### 9.1 保存

以下を保存できるようにする。

1. 統合JSON保存

   - 現在の内部データを1つの完全JSONとして保存
2. 断片JSON保存

   - 現在選択中カテゴリのみ保存
   - 例:
     - `areas.fragment.json`
     - `spots.fragment.json`
     - `characters.fragment.json`
     - `scenario.escape_manor.fragment.json`

### 9.2 読み込み

以下を読み込めるようにする。

1. 完全JSON読込

   - 全体再構築
2. 断片JSON読込

   - メッセージ欄に読み込んでから、ユーザーが「マージ」を押す流れでもよい
   - または直接マージでもよいが、今回の方針では一度 textarea に表示されることを推奨する

## 10. scenario / quest 起動 effects の追加

現状は `initial_state.active_scenario_id` や `active_quest_ids` 前提の自動開始寄りになっている。
これに加えて、action / choice の effects で以下を扱えるようにする。

- start_scenario: string
- start_quest: string
- set_primary_quest: string
- complete_quest: string

### 10.1 最低限の挙動

#### start_scenario

- `state.active_scenario_id = value`
- 必要に応じて `current_event_id` は次の遷移で解決

#### start_quest

- `active_quest_ids` に追加（重複なし）
- `quest_states[questId] = { status: "active", progress_flags: [], completed_at: null }` を補完

#### set_primary_quest

- `primary_quest_id = value`
- `current_quest_id = value`

#### complete_quest

- `quest_states[questId].status = "completed"`
- `quest_states[questId].completed_at` を current time ベースで記録（難しければ null でも可）

### 10.2 実装箇所

`applyEffects()` の中で安全に追加すること。
既存の `effects.set` や `effects.add` はそのまま残すこと。

## 11. 想定ユースケース

### ケース1

ユーザーが「村人Aとの会話イベント」をAIに作らせる
-> AIが quest開始用 action を含む scenario/event 断片を返す
-> ユーザーが scriptInput に貼る
-> 「断片マージ」
-> 統合JSONへ反映
-> そのまま実行できる

### ケース2

ユーザーが spots だけAIに作らせる
-> AIが `{"spots": {...}}` を返す
-> 貼り付け
-> マージ
-> world.spots に追加される

### ケース3

ユーザーが統合済みJSONを保存し、後日再読込する
-> 完全JSONとして読込
-> エディタとランナー両方で使える

## 12. 実装上の重要方針

- 既存の `app.js` を全面破壊しない
- 可能な範囲で editor 系関数を追加する形にする
- `normalizeScriptStructure()` と `validateScript()` を再利用する
- scriptInput を「AIとの受け渡しの正面玄関」として残す
- 自動取り込みではなく、必ずユーザー確認を介在させる
- UIは最低限でよい。まず round-trip と merge が壊れないことを優先する

## 14. 最低受け入れ条件

1. 既存サンプルJSONを読み込んで、基本エディタでカテゴリ別に編集できる
2. 編集結果を統合JSONとして textarea に戻せる
3. その JSON をそのまま「このJSONで開始」で実行できる
4. AI用プロンプトに editor 系モードが増えている
5. scriptInput に貼った断片JSONをマージできる
6. 統合JSONと断片JSONの両方を保存できる
7. 統合JSONと断片JSONの両方を読み込める
8. start_scenario / start_quest / set_primary_quest / complete_quest が動作する
9. 既存サンプルシナリオが壊れない

## 15. 補足意図

この改修の本質は「AIに全部を書かせる」ことではない。
「AIを部品作成の補助者として使い、人間が貼り付け・確認・統合する」制作フローを作ることが目的である。

したがって、

- AI生成物は必ずメッセージ欄を経由する
- マージは明示操作で行う
- 最終成果物は人間が確認した統合JSONである
  という前提を保つこと。
